1. Introduction

1.1. Purpose

The Competitive Programming Primer aims to help competitive programmers as well as computer science students and educators.

 

Computer science students and educators should have access to a simple to use software made specifically for the purpose of visualizing different data structures and algorithms, as this would make the process of learning these more accessible. Moreover, competitive programmers should have access to a curated list of past ICPC problems so they may practice to improve. This curation would be innovative as it would include difficulty sorting and techniques required for each problem. We also aim to create an easily available hackpack to be used during programming competitions.

 

1.2. Scope

We aim to deliver three software products: Visualizer, ICPC Problems Database, and Hackpack.

 

The Visualizer should provide computer science students and educators with the ability to visualize algorithms and data structures. It should be available as either a downloadable software or an application running in web browsers. Several example data structures should be readily available for anyone to view but additional functionality would provide users with the ability to build their own visualizations of advanced data structures without great hassle.

 

The ICPC Problems Database would provide access to curated past problems that have been presented in International Collegiate Programming Contests (ICPC). This database would be available to access through a browser and allow users to sort and find problems based on their difficulty and required techniques.

 

The Hackpack should be a readily available downloadable file that would include common techniques required in programming competitions as well as sample code for useful data structures or algorithms, such that it is easy to manipulate and use to solve problems. These would be available in python (for faster implementation), java (for faster execution) and pseudo-code (for general cases).

 

1.3. Definitions, Acronyms, and abbreviations

International Collegiate Programming Contest (ICPC): an international programming competition.

 

Hackpack: a collection of useful information to use as reference during programming competitions

2. Overall Description

2.1. Product perspective

The Competitive Programming Primers main parts of the ICPC database, Manim visualizer, and hackpack are independent and self-contained. Although the separate projects make up the entirety of the Primer, there are no interrelated systems that interact with each other. In regards to the database, there was an existing database which grants users access to every past ICPC problems in competitions, but due to lack of maintenance, there no longer exists such a database. Manim is also an existing visualization software, but is not user-friendly and cumbersome due to lacking a GUI, and our goal is to make it more user-friendly. As for the hackpack, it is widely used by many universities and institutions, so the idea is not new, but, as of the Fall 2025 Semester, Florida Institute of Technologys ICPC club does not have one.

2.2. Product functions

There are three main parts to the project: the collection of ICPC problems, the hackpack, and the datastructures visualizer.

 

The functions of the collection of ICPC problems is that users will be able to search for past ICPC problems with a specific difficulty, and sort a list of past ICPC problems by difficulty. And filter for or filter out ones they have not solved or ones that have specific techniques for solving them.

 

The hack pack will consist of a variety of data structures and algorithms for use in ICPC, while this document will lay out a few of such algorithms and data structures. We will add more as we see the most common algorithms and data structures used in past ICPC problems. The minimal functions of the hack-pack is that it will have at least implementations for the following data structures: linked lists, heaps, and AVL trees.

 

Along with these data structures it will also have implementations of algorithms, at minimum it will have: Dijkstra's shortest path, brute force with dynamic programing, max flow. It should also be noted that for each of these algorithms and data structures we will also include implementations for: python, java, and C. every one of these algorithms and data structures will be designed in such a way as to make them easy to use for any of ICPC problems that would require that data structure or algorithm.

 

For the data structures visualizer we will use manim to make an easy way to visualize algorithms and data structures. This visualization tool will work by proving a substitute for arrays, nodes, and pointers. For arrays the substitutes will be similar enough to the original that all someone will have to do to replace an array with ours one will only have to replace any array declarations with a constructor for our substitute and pass in the array it is substituting for as a parameter. After doing this our visualizer will display the array and will update it accordingly when the user modifies it. Pointers will work similar to arrays however assignment will also need to be replaced with our version. Lastly for nodes all users will need to do is make there old nodes inherit from ours and use our getter and setter for getting and setting the displayed value and the child pointer(s)

2.3. User characteristics

Each of the three main parts of the project have a different target demographic.

 

For the collection of ICPC problems the target demographic is coders with a little bit of coding experience (ideally having passed CSE 1002) and who want to compete in an upcoming ICPC. These users are expected to understand how to navigate the collection of ICPC problems though that is not to say we will not try to make it easier to find stuff. The users are expected to be able to read and understand what a given ICPC is asking. The users are expected to be able to fix their code when given an example that breaks it. The user is expected to be able to implement algorithms from an abstract description.

 

For the hack pack the target demographic is coders with a medium amount of coding experience (ideally having passed algorithms and data structures) and are competing in an upcoming ICPC. We expected the user to have heard of each of the algorithms and data structures included and knowing when and how to use them.

 

For the algorithms and data structures visualizer the target audience is teachers for some sort of algorithms and data structures course. It is expected these teachers will already have a python implementation for any algorithms and data structures they wish to teach. They are also expected to understand this implementation on the level of someone who has created it. They are also expected to have strong debugging skills as if they improperly integrate our tool into their algorithms or data structure they are expected to be able to find and fix the problem themself

2.4. Constraints

Each of the three main parts of the project have different target constraints.

 

For the collection of ICPC problems we are not able to use any paid tools for hosting the database. It is impossible to automatically categorize each problem so all such categorizations must be done by a human. When sorting the problems we can not guarantee categorizing by everything the user may want and do to the large quantity we can not guarantee a 100% error free collection.

 

For the hack-pack we are not allowed to make it bigger than a one foot by one foot box. We are also not going to be able to get every algorithm and data structure that might appear in a competition in the hack-pack as there are many algorithms and data structures out there.

 

For the visualization tool we can not guarantee full customization over how the information is presented. We can not guarantee that we will have integration with all of the features python structures come with as there are many functions built into python. For example something like the sorted function only works on arrays so it will work on our variation. We can not guarantee we will be able to help you visualize anything not purely constructed of arrays, nodes, and pointers.

2.5. Assumptions and dependencies

We do not have that many assumptions in this project. For the collection of ICPC problems we are assuming that the locations we believe they are in do not pose any difficulties while trying to draw from them. We are also assuming that we will be able to host a database in an easily accessible location. For the hack-pack the only assumption is that we will be able to code up the algorithms and data structures we wish to include.

 

For the visualization tool we have already verified we can draw circles and arrows and text wherever we want but we are assuming we can also draw squats and curved arrows. We are also assuming that it will be easy to stop elements from overlapping on screen

 

For dependencies each of the three main parts are fully independent of one another however within each part there are still some dependencies. For the curation of ICCPC problems we have to collect them before we can start on curation though we can make the database separately on example data and use the real stuff when we get it. For the visualization tool we have to be able to draw the shapes before we worry about preventing intersections.
