1.2 Scope

The purpose of this document is to overview the testing that the Competitive Programming Primer features must undergo. These features include the Visualizer, the ICPC Problem Database, and the Hackpack.

The Visualizer will require testing in terms of user input and the output of the program. Should any of the output display incorrect information or have display issues, that would be a failed case.

The database will be tested on how correct the data stored is and on issues with accessibility to this data.

The Hackpack will be tested on the correctness and usefulness of the provided information, as well as different users ability to access and download it.

Test Strategy and Approach:

 

For each of the three parts of the project there is a different way to test it.

For the collection of ICPC problems we can test if the collection holds all of them by counting how many we have and counting how many the ICPC has. To test that they are an easily accessible location we can look at the average number of key presses and mouse clicks required to get to a specific problem. To test the accuracy of the sorting we can solve a few and see if the tactics we said would have to be used were used. We will also test each sorting feature individually by attempting to apply that filter and comparing that result to what we expect from our knowledge of what we entered into the database

For the hack-pack we can test each one of the algorithms and data structures in it by finding an ICPC problem from the collection that we think uses those algorithms or data structures and see if our implementation can be used to correctly solve it. Also we can test its usefulness by seeing if the competitive codding club would let us try giving them to people during one of their practice competitions

For the visualizer we can test it by taking some of the algorithms and data structures made in the hack-pack and attempting to use the visualizer to visualize them.

 

 

 

Entry Criteria:

 

We can begin testing the collection of ICPC problems once we have a database with at least three ICPC problems on it since we need to have enough problems to properly test sorting and filtering features.

For the hack-pack we can start testing each algorithm and data structure once the ICPC problem curation has found a problem they believe would require it.

For the visualizer we can test it once one of the problems from the hack-pack passes its test

 

 

Exit Criteria:

 

For the collection of ICPC problems we can stop testing once we have all the ICPC problems and try each of the possible sorting and filtering options on it.

For the hack-pack we can safely stop testing once all of the algorithm and data structures implemented have been used to solve at least one ICPC or kattis problem since to pass one of these problems the code is subjected to a robust and varied set of test cases.

For the visualizer we can safely stop testing once we have successfully used it to visualize all the problems in the hack-pack and none of them had an excessive amount of intersecting lines.
